## Code Review — PR #12 (re-review after commits `7296ae5 / e4102f5 / 637f391 / 6de6f20`)

### Summary

This PR implements a Fixed-Term Deposit Vault Smart Contract with 31 unit tests. Since the previous review, two MINORs were correctly resolved (denomination check via `posting.denomination` and `Write(*)` scope restriction). However, **two MAJOR issues flagged in the previous review remain unresolved**, and a **new MAJOR issue was introduced** in the `post_posting_hook` fix: the penalty amount is credited back to the customer's `DEFAULT` address instead of being truly forfeited.

**Overall verdict: REQUEST CHANGES**

---

### Architecture & Design

The dispatcher pattern (`scheduled_event_hook` → `_handle_daily_accrual` / `_handle_maturity_event`) is clean and testable. Balance address model (`DEFAULT` / `ACCRUED_INTEREST`) and the double-entry `CustomInstruction` pattern are correct throughout. No structural regressions introduced in this pass.

---

### Specific Issues

---

**File**: `contracts/fixed_term_deposit.py` (lines 116–133) — **Severity: MAJOR** *(unresolved from previous review)*

`accrued_interest` and `days_to_maturity` are still declared as regular `Parameter` objects with `derived=True`. `DerivedParameter` is not imported and there is no `derived_parameters` module-level list. The `derived=True` kwarg is not a Vault API 4.0 construct — Vault will not route calls to `derived_parameter_hook` for these values, and the hook will silently never be invoked in a live environment.

Required fix:
```python
from contracts_api import DerivedParameter   # add to imports

# Remove the two Parameter blocks with derived=True from the parameters list.
# Add at module level, after the parameters list:
derived_parameters = [
    DerivedParameter(name="accrued_interest", shape=NumberShape(), display_name="Accrued Interest"),
    DerivedParameter(name="days_to_maturity",  shape=NumberShape(), display_name="Days to Maturity"),
]
```

---

**File**: `contracts/fixed_term_deposit.py` (lines 199, 381) — **Severity: MAJOR** *(unresolved from previous review)*

`.date()` is still called on `opt_maturity.value` in `activation_hook` (line 199) and `derived_parameter_hook` (line 381). In real Vault, `DateShape` returns a `datetime.date` object, not a `datetime`. Calling `.date()` on a `date` raises `AttributeError`. Tests pass only because `make_vault` passes `datetime` objects as `maturity_date`, masking the runtime divergence.

```python
# line 199 — WRONG (works in tests because mock passes datetime, not date)
if maturity_date.date() <= effective_date:

# line 381 — WRONG (same reason)
days = max((opt_maturity.value.date() - today).days, 0)
```

Required fix — drop `OptionalShape` on `maturity_date`, use plain `DateShape()`, remove all `.date()` calls and `.is_set()` / `.value` unwrapping. Update tests to pass `FUTURE_MATURITY.date()` (a `date` object) instead of the `datetime` constants.

---

**File**: `contracts/fixed_term_deposit.py` (lines 323–349) — **Severity: MAJOR** *(new)*

The `post_posting_hook` credits the forfeited `penalty` amount to `DEFAULT` (the customer's own account), making the penalty financially meaningless. With `penalty_rate=0.20` and `accrued=£10`, the customer receives both £8 (net payout) and £2 (penalty) in `DEFAULT`, for a total of £10 — identical to what they would have received with no penalty at all. The description `"Early closure penalty forfeiture"` is misleading: nothing is actually forfeited from the customer's perspective.

```python
# lines 335–343 — credits penalty back to customer's DEFAULT (wrong)
Posting(
    credit=True,
    amount=penalty,
    account_address=DEFAULT_ADDRESS,   # ← customer gets penalty back
    ...
),
```

Required fix: credit the penalty to an internal/contra address (e.g. `PENALTY_INCOME`) rather than `DEFAULT`. If the contract does not track a separate income address, at minimum document and justify this decision explicitly; do not label it "forfeiture."

---

**File**: `tests/test_fixed_term_deposit.py` (line 25) — **Severity: MINOR**

```python
import sys, os
sys.path.insert(0, os.path.join(os.path.dirname(__file__), ".."))
```

`sys` and `os` are fine in test code (not contract code), but `sys.path` manipulation is fragile and unnecessary if the SDK is installed correctly via `pip install -e .` or `PYTHONPATH`. The other test files (`test_savings_product.py`, `test_current_account.py`) should be checked to see if they use the same pattern; if so, a `conftest.py` at repo root or a proper `pyproject.toml` entry point would be cleaner.

---

**File**: `tests/test_fixed_term_deposit.py` (lines 355–368, `test_early_closure_penalty_applied`) — **Severity: MINOR**

This test asserts `len(result.posting_instructions_directives) == 1` and inspects only `posting_instructions[0]` (first `CustomInstruction`). With the current two-CI structure, it passes because the first CI is the payout. However, the test does not assert that the penalty CI's destination is correct — it only checks the payout CI. As a result the financial bug in Issue 3 above is not caught by this test. A separate assertion checking the penalty CI's `account_address` would catch the regression.

### What Was Fixed

1. **MINOR resolved** — denomination check now uses `posting.denomination` directly (correct) instead of `coords[0].denomination` (fragile).
2. **MINOR resolved** — `.claude/settings.json` `Write(*)` replaced with scoped `Write(contracts/*)`, `Write(tests/*)`, `Write(ai-specs/*)`.
3. **MAJOR resolved** — `ACCRUED_INTEREST` residual balance: both `net_payout` and `penalty` are now debited from `ACCRUED_INTEREST` via separate `CustomInstruction` objects, zeroing the address in full. The test `test_early_closure_accrued_interest_zeroed` correctly verifies this.

---

### Testing

31 tests, up from 22 planned. AAA pattern is consistent. `client_transactions={}` passed in all `PrePostingHookArguments` and `PostPostingHookArguments` calls. `ZoneInfo("UTC")` used throughout — no `timezone.utc`. Test isolation is good (each test builds its own mock vault).

Accuracy gap that persists: `make_vault` wraps `datetime` objects in `OptionalValue(maturity_date)` (line 76), but real Vault's `DateShape` returns `datetime.date`. After fixing Issue 2, `FUTURE_MATURITY`, `TODAY_MATURITY`, and `PAST_MATURITY` must be `date` objects (e.g., `datetime(2025, 6, 30, tzinfo=UTC).date()`), or the activation tests will begin failing with `AttributeError` in a live Vault instance.

---

### Security

No hardcoded secrets. No stdlib imports in contract code. No mutable global state. No network calls. No issues.

---

### Final Verdict

**REQUEST CHANGES**

Three blocking issues before merge:

1. **`derived_parameters`** — replace `derived=True` on `Parameter` with the `DerivedParameter` class in a separate `derived_parameters` list; import `DerivedParameter` from `contracts_api`.
2. **`.date()` on `DateShape` value** — `DateShape` returns `datetime.date`; the `.date()` call at lines 199 and 381 raises `AttributeError` in a live Vault instance. Fix the contract and update test fixtures to pass `date` objects.
3. **Penalty credited to `DEFAULT`** — the forfeited penalty amount is returned to the customer's own account, neutralising the penalty entirely. Credit it to a `PENALTY_INCOME` internal address, or justify explicitly if the MVP intentionally skips income tracking.
