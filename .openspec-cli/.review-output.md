**File**: `contracts/fixed_term_deposit.py` (lines 117–132) — **Severity: MAJOR** *(unresolved from previous review)*

`derived_parameters` are still declared as regular `Parameter` objects with `derived=True`:

```python
Parameter(
    name="accrued_interest",
    shape=NumberShape(),
    level=ParameterLevel.INSTANCE,
    derived=True,          # ← non-standard; undefined runtime behavior
    ...
),
```

`DerivedParameter` is not imported and a `derived_parameters` module-level list does not exist. The `derived=True` flag on a standard `Parameter` is not a documented API 4.0 construct — Vault will not know to route these to `derived_parameter_hook`. The comment added in the fix commit ("API 4.0 SDK pattern") is incorrect; this is not the SDK pattern.

**Required fix**:
```python
from contracts_api import DerivedParameter  # add to imports

# Remove the two Parameter blocks with derived=True from parameters list
# Add at module level, after parameters:
derived_parameters = [
    DerivedParameter(name="accrued_interest", shape=NumberShape(), display_name="Accrued Interest"),
    DerivedParameter(name="days_to_maturity",  shape=NumberShape(), display_name="Days to Maturity"),
]
```

**File**: `contracts/fixed_term_deposit.py` (lines 86–88, 195, 349) — **Severity: MAJOR** *(unresolved from previous review)*

`maturity_date` still uses `OptionalShape(shape=DateShape())`. The `is_set()` guard added in the fix commit addresses the `None` crash, but the root `.date()` call on lines 195 and 349 is still present:

```python
# activation_hook line 195
if maturity_date.date() <= effective_date:   # maturity_date is already a date object

# derived_parameter_hook line 349
days = max((opt_maturity.value.date() - today).days, 0)
```

In a real Vault instance, `DateShape` returns a `datetime.date` object (not `datetime`). Calling `.date()` on a `date` object raises `AttributeError`. Tests pass only because `make_vault` mocks `maturity_date` with a `datetime`, masking the type divergence.

**Required fix** — two options:

*Option A (recommended — matches plan)*: Change to `DateShape()`, drop `OptionalShape`, remove `.value` unwrapping and `.date()` calls:
```python
Parameter(name="maturity_date", shape=DateShape(), level=ParameterLevel.INSTANCE, ...)

# activation_hook:
maturity_date  = vault.get_parameter_timeseries(name="maturity_date").latest()
effective_date = hook_arguments.effective_datetime.date()
if maturity_date <= effective_date: ...
```

*Option B*: Keep `OptionalShape` but remove `.date()` calls since `.value` already yields a `date`:
```python
maturity_date = opt_maturity.value          # already datetime.date
if maturity_date <= effective_date: ...     # direct comparison, no .date()
```

Also update `make_vault` in tests to pass a `datetime.date` (not `datetime`) when `maturity_date` is set, to match real Vault behavior.

**File**: `contracts/fixed_term_deposit.py` (lines 238–244) — **Severity: MINOR** *(partially improved)*

The denomination check was narrowed to check only `coords[0]`:

```python
coords = list(posting.balances().keys())
if coords and coords[0].denomination != denomination:
```

This is better than iterating all coordinates, but still derives denomination from a balance coordinate rather than from the posting instruction itself. A posting instruction object in Vault's API carries a denomination attribute directly — using that is more semantically correct and avoids relying on balance dictionary ordering. If the mock in tests returns a deterministic key order this will not surface in tests, but in production the `list()` of a dict is unordered prior to Python 3.7 semantics on SDK internals.
