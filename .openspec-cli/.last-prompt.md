## Domain Layer Rules

### Models
- Pure Python dataclasses or `@dataclass` classes — zero framework dependencies
- Business logic lives inside domain methods
- Use `__init__` or classmethods as factory methods to enforce invariants

```python
from dataclasses import dataclass

@dataclass
class Candidate:
    id: int | None
    first_name: str
    email: str

    @classmethod
    def create(cls, first_name: str, email: str) -> "Candidate":
        if "@" not in email:
            raise InvalidEmailError(email)
        return cls(id=None, first_name=first_name, email=email)
```

### Repository Interfaces
- Abstract base classes in the domain layer
- Implementations live in infrastructure
- Return domain models, not SQLAlchemy ORM objects

```python
from abc import ABC, abstractmethod

class CandidateRepository(ABC):
    @abstractmethod
    async def find_by_id(self, id: int) -> Candidate | None: ...

    @abstractmethod
    async def save(self, candidate: Candidate) -> Candidate: ...

    @abstractmethod
    async def exists_by_email(self, email: str) -> bool: ...
```

## Application Layer Rules

- Services receive repository interfaces via constructor injection
- Accept Pydantic DTOs — never raw `dict` from request
- Raise domain exceptions for business rule violations
- No SQLAlchemy or FastAPI imports in this layer

```python
class CandidateService:
    def __init__(self, repo: CandidateRepository) -> None:
        self._repo = repo

    async def create(self, dto: CreateCandidateRequest) -> CandidateResponse:
        if await self._repo.exists_by_email(dto.email):
            raise DuplicateEmailError(dto.email)
        candidate = Candidate.create(dto.first_name, dto.email)
        saved = await self._repo.save(candidate)
        return CandidateMapper.to_response(saved)
```

## Presentation Layer Rules

- Routers are thin — validate via Pydantic, call service, return response
- Use FastAPI `Depends()` for dependency injection of services
- HTTP status codes: 201 for POST create, 200 for GET/PUT, 204 for DELETE
- Register global exception handlers on the FastAPI `app` instance

```python
router = APIRouter(prefix="/candidates", tags=["Candidates"])

@router.post("/", status_code=201, response_model=CandidateResponse)
async def create_candidate(
    request: CreateCandidateRequest,
    service: CandidateService = Depends(get_candidate_service),
) -> CandidateResponse:
    return await service.create(request)
```

### Global Exception Handlers
```python
@app.exception_handler(DomainException)
async def domain_exception_handler(request: Request, exc: DomainException):
    return JSONResponse(status_code=exc.http_status,
        content={"success": False, "code": exc.code, "message": str(exc)})

@app.exception_handler(RequestValidationError)
async def validation_exception_handler(request: Request, exc: RequestValidationError):
```

# Python + FastAPI — Standards and Best Practices

## Technology Stack
- Python 3.12+ — type hints mandatory on all functions and methods
- FastAPI with Pydantic v2
- SQLAlchemy 2.x (async) + PostgreSQL
- Alembic for migrations (scripts under `alembic/versions/`)
- pytest + httpx for testing (90% coverage threshold)
- Ruff for linting + mypy (strict) for type checking

## Architecture
```
domain/        Pure Python — no FastAPI or SQLAlchemy dependencies
application/   Services + Pydantic DTOs + Mappers
presentation/  FastAPI Routers + Depends() wiring + Exception handlers
infrastructure/ SQLAlchemy repositories + DB session + Config
```

## Key Rules
- Type hints required on every function signature — mypy strict enforced
- Pydantic models validate all external input before reaching services
- SQLAlchemy ORM models live only in `infrastructure/` — never imported in domain or application
- Alembic for all schema changes — never use `Base.metadata.create_all()` in production
- FastAPI `Depends()` for dependency injection — no manual service instantiation in handlers
- Global exception handlers registered on the `app` instance for domain and validation errors

## Naming Conventions
| Element | Convention | Example |
|---|---|---|
| Files / Modules | snake_case | `candidate_service.py` |
| Classes | PascalCase | `CandidateService` |
| Functions / vars | snake_case | `create_candidate` |
| DB tables | snake_case | `work_experience` |
| Alembic scripts | auto-named | `abc123_add_resume_column.py` |

## Testing
- pytest with `pytest-asyncio` for async tests
- httpx `AsyncClient` for HTTP integration tests
- `pyproject.toml` enforces 90% coverage via `pytest --cov`
- AAA pattern; test name: `test_should_do_something_when_condition`

## Git Workflow
- Branch naming: `feature/<ticket-id>-backend`
- Commits: English, conventional commits

## Implementation Instructions
# develop-backend

Analyze and implement the Jira ticket: KAN-6

## Pre-flight checklist (run before writing any code)

1. Read `openspec/config.yaml` and resolve:
   - Active stack (`stack:` key)
   - Stack agent path (`stacks.<stack>.agent`)
   - Stack standards path (`stacks.<stack>.standards`)
   - Tooling commands: `build_command`, `test_command`, `run_command`, `coverage_command`
2. Load and internalize the resolved stack agent and standards files
3. Read `ai-specs/specs/base-standards.mdc`
4. Read `ai-specs/specs/documentation-standards.mdc`
5. If a plan already exists at `ai-specs/changes/<ticket-id>_backend.md`, read it before starting

## Implementation steps

### Step 0 — Create feature branch
```bash
git checkout main && git pull origin main
git checkout -b feature/<ticket-id>-backend
git branch  # verify
```

### Step 1 — Understand the ticket
- Use Jira MCP to fetch ticket details if available; otherwise read from the plan file
- Identify all affected layers: Domain, Application, Presentation, Infrastructure

### Step 2 — Write tests first (TDD)
- Write failing unit tests for the business logic **before** any implementation
- Write failing controller slice tests for the new endpoint **before** the controller exists
- Run tests and confirm they fail for the right reason:
  ```bash
  pytest
  ```

### Step 3 — Implement to make tests pass
- Follow the implementation order defined in the plan file
- Use only the tooling commands from `openspec/config.yaml` for the active stack
- Never hardcode build tool commands (e.g. `./gradlew`, `npm`, `pytest`) — always use the resolved variable

### Step 4 — Verify build and coverage
```bash
pip install -r requirements.txt
pytest --cov --cov-report=html
```
Coverage must be >= 90%. If it is not, add tests before continuing.

### Step 5 — Manual endpoint testing
Test all HTTP status codes for every new or modified endpoint:
- Happy path (200 / 201)
- Validation error (400)
- Not found (404)
- Conflict (409) — if applicable
- Server error (500) — if applicable

Document the curl commands used in a comment block in the plan file.

### Step 6 — Update technical documentation
Run the update-docs command:
```
/update-docs
```
Minimum files to review and update:
- `ai-specs/specs/data-model.md` — if schema changed
- `ai-specs/specs/api-spec.yml` — if endpoints changed
- `ai-specs/specs/stacks/<stack>-standards.mdc` — if libraries or patterns changed

### Step 7 — Commit and open PR
Stage only the files affected by this ticket:
```bash
git add <affected files only>
```
Use a descriptive commit message (conventional commits format, English):
```
feat(<scope>): <short imperative description>

- <what changed and why>
- Resolves <ticket-id>
```
Push and open a PR using GitHub CLI:
```bash
git push -u origin feature/<ticket-id>-backend
gh pr create --title "[<ticket-id>] <summary>" --body "<description>"
```

## Rules
- **Never skip Step 0** (branch) or Step 6 (docs)
- **TDD is mandatory** — tests before implementation, always
- **Use resolved tooling commands** from `openspec/config.yaml` — never hardcode
- All code, comments, and commit messages must be in English
- Do not commit unrelated files; stage only what belongs to this ticket
