## API 4.0 — Critical changes from 3.x

These differences cause failures with the installed SDK. Apply them automatically.

### 1. ZoneInfo required (not datetime.timezone)
```python
from zoneinfo import ZoneInfo
dt = datetime(2024, 1, 1, tzinfo=ZoneInfo("UTC"))   # CORRECT
dt = datetime(2024, 1, 1, tzinfo=timezone.utc)       # WRONG → InvalidSmartContractError
```

### 2. Phase is on BalanceCoordinate (key), not on Balance (value)
```python
# CORRECT — iterate items(), phase is on coord
for coord, balance in posting.balances().items():
    if coord.phase == Phase.COMMITTED:
        total += balance.net

# WRONG — Balance has no .phase attribute
for balance in posting.balances().values():
    if balance.phase == Phase.COMMITTED:   # AttributeError
```

### 3. Rejection not Rejected, return not raise
```python
# CORRECT API 4.0
return PrePostingHookResult(
    rejection=Rejection(
        message="Insufficient funds",
        reason_code=RejectionReason.INSUFFICIENT_FUNDS,
    )
)

# WRONG — API 3.x pattern
raise Rejected("Insufficient funds", reason_code=RejectedReason.INSUFFICIENT_FUNDS)
```

### 4. CustomInstruction has no client_transaction_id
```python
# CORRECT — use instruction_details for traceability
CustomInstruction(
    postings=[...],
    instruction_details={
        "description": "Monthly interest accrual",
        "hook_execution_id": str(vault.get_hook_execution_id()),
    },
)

# WRONG — doesn't exist in API 4.0
CustomInstruction(postings=[...], client_transaction_id="APPLY_INTEREST_123")
```

### 5. PrePostingHookArguments requires client_transactions
```python
# CORRECT
PrePostingHookArguments(
    effective_datetime=dt,
    posting_instructions=[posting],
    client_transactions={},    # ← required in API 4.0
)
```

### 6. get_balances_observation not get_balance_timeseries
```python
# CORRECT API 4.0
balances = vault.get_balances_observation(fetcher_id="live_balances").balances

# WRONG — API 3.x pattern
balances = vault.get_balance_timeseries().latest()
```

## Hook signatures (API 4.0)

```python
def activation_hook(vault, hook_arguments: ActivationHookArguments) -> ActivationHookResult:
    return ActivationHookResult(scheduled_events_return_value={})


def pre_posting_hook(vault, hook_arguments: PrePostingHookArguments) -> PrePostingHookResult:
    denomination = vault.get_parameter_timeseries(name="denomination").latest()
    balances     = vault.get_balances_observation(fetcher_id="live_balances").balances
    current      = _get_committed_balance(balances, denomination)

    # Phase is on the key (BalanceCoordinate), not on the Balance value
    posting_effect = Decimal("0")
    for posting in hook_arguments.posting_instructions:
        for coord, balance in posting.balances().items():
            if coord.phase == Phase.COMMITTED:
                posting_effect += balance.net

    if current + posting_effect < Decimal("0"):
        return PrePostingHookResult(
            rejection=Rejection(
                message=f"Insufficient funds: balance {current} {denomination}",
                reason_code=RejectionReason.INSUFFICIENT_FUNDS,
            )
        )
    return PrePostingHookResult()


def post_posting_hook(vault, hook_arguments: PostPostingHookArguments) -> PostPostingHookResult:
    return PostPostingHookResult(posting_instructions_directives=[])


def scheduled_event_hook(
    vault, hook_arguments: ScheduledEventHookArguments
) -> ScheduledEventHookResult:
    return ScheduledEventHookResult(
        posting_instructions_directives=[],
        update_account_event_type_directives=[],
    )
```

### Specific Issues

---

**File**: `contracts/fixed_term_deposit.py` (line 41) — **Severity: CRITICAL**

```python
from contracts_api.utils.exceptions import InvalidContractParameter
```

The Vault sandbox only permits imports from the top-level `contracts_api` package. Importing from a submodule path (`contracts_api.utils.exceptions`) is not listed as allowed and will likely cause a runtime rejection when the contract is loaded. The stack agent spec is explicit: *"Only `contracts_api` and `decimal` imports allowed in contract code."*

**Fix**: Check whether `InvalidContractParameter` is exported from the top-level package:
```python
from contracts_api import InvalidContractParameter
```
If it is not re-exported at the top level, the activation validation should be restructured to not depend on this exception class. An alternative is to let the `activation_hook` return a rejection via `ActivationHookResult` with a specific rejection, though the plan prescribes `InvalidContractParameter`.

---

**File**: `contracts/fixed_term_deposit.py` (lines 117–132) — **Severity: MAJOR**

The implementation put derived parameter declarations inside the `parameters` list using `derived=True` on regular `Parameter` objects:

```python
Parameter(
    name="accrued_interest",
    shape=NumberShape(),
    level=ParameterLevel.INSTANCE,
    derived=True,         # ← non-standard
    ...
),
```

The plan and the agent spec both define a separate module-level `derived_parameters` list using `DerivedParameter`:

```python
from contracts_api import DerivedParameter   # ← missing from imports

derived_parameters = [
    DerivedParameter(name="accrued_interest",  shape=NumberShape(), display_name="Accrued Interest"),
    DerivedParameter(name="days_to_maturity",  shape=NumberShape(), display_name="Days to Maturity"),
]
```

`DerivedParameter` is not imported anywhere in the contract. The `derived=True` flag on a regular `Parameter` is not a documented API 4.0 pattern and its runtime behavior is undefined.

**Fix**: Remove the two `Parameter` blocks with `derived=True` from `parameters`, import `DerivedParameter`, and add a `derived_parameters` list at module level.

---

**File**: `contracts/fixed_term_deposit.py` (lines 85–88, 192, 195, 350) — **Severity: MAJOR**

The `maturity_date` parameter uses `OptionalShape(shape=DateShape())` instead of `DateShape()` directly (as specified in the plan). This introduces a compound issue:

1. In `activation_hook` line 192: `.latest().value` is required to unwrap the `OptionalValue` — but if the parameter is unset, `.value` is `None` and `None.date()` on line 195 throws `AttributeError` with no guard.
2. In real Vault, a `DateShape` parameter returns a `datetime.date` object from `.latest()`. The code calls `.date()` on that value (lines 195, 350), which does not exist on `date` objects. Tests pass because `make_vault` mocks it with a `datetime` — but runtime behavior diverges.

**Fix**: Change the parameter shape back to `DateShape()` as planned. Remove `OptionalShape` wrapper:
```python
Parameter(
    name="maturity_date",
    shape=DateShape(),
    level=ParameterLevel.INSTANCE,
    ...
)
```
Then in `activation_hook`, call `.latest()` directly (no `.value`) and remove the `.date()` call since the result is already a `date`:
```python
maturity_date  = vault.get_parameter_timeseries(name="maturity_date").latest()
effective_date = hook_arguments.effective_datetime.date()
if maturity_date <= effective_date: ...
```

**File**: `contracts/fixed_term_deposit.py` (lines 238–249) — **Severity: MINOR**

The denomination guard in `pre_posting_hook` iterates **all** balance coordinates of all postings and rejects if any coord denomination differs:

```python
for posting in hook_arguments.posting_instructions:
    for coord, _ in posting.balances().items():
        if coord.denomination != denomination:
```

A single posting instruction that has internal balance entries across multiple phases or addresses (e.g., `PENDING_OUT` phase entries) might include coordinates with unexpected denominations, causing false rejections. The guard should check the posting instruction denomination directly rather than deriving it from every balance coordinate.
